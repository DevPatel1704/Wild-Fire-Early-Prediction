## What This Pipeline Does

Reads simulated ICU patient heart rate data from a watched folder, computes average
heart rate per patient in tumbling 2-minute windows, and fires a clinical alert when
a patient's average HR stays above 100 bpm across two back-to-back windows.

```
CSV files dropped into stream_input/
        |
        v
readStream (watched directory)
        |
        v
withWatermark("timestamp", "2 minutes")
        |
        v
groupBy(window("timestamp", "2 min"), patient_id)
        |
        v
avg(heart_rate)  ->  windowed aggregation
        |
        |---> filter(avg_hr > 100)  ->  console sink   [filtered output stream]
        |
        '---> foreachBatch  ->  consecutive window check  ->  CLINICAL ALERT
```

## Written Explanation

### Why Tumbling Windows?

I used tumbling windows because the assignment asks to detect *sustained* abnormal
heart rate — not just a brief spike. A tumbling window splits time into fixed,
non-overlapping 2-minute buckets. Each reading belongs to exactly one bucket, so
each window gives a clean independent average for that period.

I considered sliding windows but decided against them for this scenario. With sliding
windows the same reading appears in multiple windows, which would make the
"two consecutive periods" check ambiguous — you could get an alert from two overlapping
windows that both contain the same bad readings, not two genuinely separate bad periods.

Tumbling windows also make the consecutive-window check straightforward: since each
window is exactly 120 seconds and they sit right next to each other, I just check
that the timestamps of two high-HR windows are 120 seconds apart (with a small
tolerance). Clean and easy to reason about.

### Where the Pipeline Requires State

State is needed in two places:

**1. Spark's internal watermark state**

When I call `.withWatermark("timestamp", "2 minutes")`, Spark has to keep partial
aggregation results in memory for every open window. It doesn't know a window is
"done" until the watermark (max event timestamp - 2 min delay) moves past that
window's end time. So for example while processing window 09:00-09:02, Spark holds
those partial sums in state until the watermark reaches 09:02. Only then does it
emit the final avg HR and clean up that state.

**2. My per-patient window history (application level)**

I maintain a Python dictionary `_window_history` on the driver that maps each
patient ID to their recent high-HR windows:

```python
_window_history: dict[int, list[tuple]]
# patient_id -> [(window_start, avg_hr), ...]
```

This is necessary because a single window result in isolation can't tell me if the
elevated HR is sustained — I need to remember what happened in the *previous* window
too. After each batch, I update this dict: if a patient's avg HR is above 100 I
append the window to their history, check if any two adjacent entries are 120 seconds
apart, and fire an alert if so. If HR drops back to normal I clear their history.
Without this cross-batch memory the consecutive-window alert would be impossible.

## Dataset

**IoMT Health Monitoring** — Kaggle  
50,000 ICU patient records with heart rate, SpO2, blood pressure, temperature,
fall detection and clinical alert flags. Heart rates range from 60 to 149 bpm;
54.6% of readings exceed the 100 bpm threshold.

For this pipeline the readings are assigned to 500 simulated patients and spread
across 10 consecutive 2-minute event-time windows (09:00 to 09:20), with 10 readings
per patient per window. This guarantees every patient appears in every window so
the consecutive-window alert logic actually fires.
